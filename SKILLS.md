# Research Repository Operating Skills

This file defines how agents should run experiments, save code/data, keep results reproducible, and prepare the repository for later paper writing and Zenodo release.

Use this file when an empty repository is used as a research workspace where agents run experiments, commit scripts and raw logs, generate reports, and later write a paper that cites the GitHub repository as the reproducibility source.

## Core principle

```text
No code + no raw log = no result.
```

Markdown tables, summaries, and claims are not enough. A result becomes a result only when the repository contains:

```text
1. committed generator script
2. fixed seed/config
3. committed raw log or equivalent reproducibility artifact
4. generated report pointing to the script and raw log
5. status label in results/STATUS.md
6. passing raw-log reproducibility check when applicable
```

## Layer separation

Always separate these layers.

```text
classical-effective component layer:
  phenomenological or classical models used to build stable reusable parts
  examples: membrane, source-sink, reservoir, converter, road, buffer

quantum/audit witness layer:
  tests that ask whether a substructure needs a quantum witness
  examples: negativity, entanglement, dephase sensitivity, measurement backaction

paper/narrative layer:
  interpretation, discussion, limitations, and final manuscript text
```

Do not let the witness layer interrupt component development unless a component is being promoted as quantum-specific.

### Correct use of classical results

```text
classical-effective result = useful component if stable and reproducible
```

Do not treat classical explanation as failure. It is only a failure if the file claims a quantum-specific result.

### Boundary-guard rule

For component files, keep vocabulary classical unless quantum controls are explicitly present.

Allowed component-layer words:

```text
classical
phenomenological
stochastic
component
throughput
selectivity
retention
release
conversion
fidelity
promiscuity
gating
hysteresis
poisoning
stress
buffering
source
sink
reservoir
converter
membrane
road
terrain
```

Quantum/audit words require explicit controls and should usually be isolated in audit files:

```text
quantum
negativity
entanglement
coherence
dephase
measurement backaction
Bell
CHSH
nonclassical
quantum advantage
```

If those words appear in a component report, add a clear disclaimer or move the discussion into an audit/witness file.

## Required repository layout

Use this structure for new repositories.

```text
README.md
SKILLS.md
requirements.txt
.gitignore
.github/workflows/reproducibility.yml

scripts/
  check_raw_logs.py
  quantum_or_core_scripts.py
  phenomenology/
    component_generators.py

configs/
  experiment_name/
    seed1234_config.json

data/
  experiment_name/
    raw logs: json/jsonl/csv/npz metadata

results/
  STATUS.md
  experiment_name_YYYY-MM-DD.md

experiments/
  protocols and preregistered plans

paper/
  manuscript drafts, figures, tables, bibliography notes

archive/
  old or superseded material that should not be cited as current result
```

## Status labels

Every report in `results/` must be classified in `results/STATUS.md`.

```text
CODE_BACKED:
  generated by a committed script with a runnable command

RAW_LOG_BACKED:
  generated by a committed script and committed raw json/csv/jsonl/npz log

PHENOMENOLOGY_NOTE:
  conceptual or exploratory note; may contain illustrative tables but not a reproducible result yet

QUARANTINED_CLAIM:
  previously stated or implied result that failed audit or lacks reproduction support

META:
  repository management, index, or summary file; not an experiment result
```

Do not use `RAW_LOG_BACKED` unless `scripts/check_raw_logs.py` can verify it, or unless the report clearly explains why byte-level regeneration is impossible and what equivalent check replaces it.

## Experiment lifecycle

Follow this sequence.

### 1. Protocol first

Create an experiment plan before running large sweeps.

Path:

```text
experiments/<experiment_name>_protocol_YYYY-MM-DD.md
```

Protocol must state:

```text
question
layer: classical-effective / quantum-audit / paper-analysis
model
metrics
controls
success/failure criteria
seeds
expected raw outputs
known limitations
forbidden claims
```

### 2. Generator script

Create a script before committing numbers.

Path examples:

```text
scripts/phenomenology/converter_core.py
scripts/quantum_lattice_core.py
scripts/audit/negativity_causality.py
```

Script must:

```text
accept seed/config/output path arguments
write raw data deterministically for fixed seed
print or save machine-readable outputs
contain a docstring saying what layer it belongs to
avoid hard-coded report tables as the only source of truth
```

### 3. Raw log

Save the raw output under `data/`.

Examples:

```text
data/converter/converter_core_seed8128_summary.csv
data/membrane/membrane_sweep_seed1200.jsonl
data/quantum_core/negativity_scan_seed0.json
```

Raw logs should include or point to:

```text
script path
git commit if available
seed
config
model version
all measured metrics
all sweep settings
```

### 4. Report

Create a human-readable report under `results/` only after raw logs exist.

Report header must include:

```text
Status: RAW_LOG_BACKED or CODE_BACKED or PHENOMENOLOGY_NOTE
Generator script: <path>
Raw log: <path>
Run command: <exact command>
Layer: classical-effective / quantum-audit / paper-analysis
```

The report may summarize tables, but those tables must be derived from raw logs.

### 5. Status index

Update `results/STATUS.md` in the same commit or immediately after.

### 6. Reproducibility check

Add or update `scripts/check_raw_logs.py` so that canonical raw logs are regenerated and compared.

Run locally:

```bash
python scripts/check_raw_logs.py
```

If it fails, do not mark the result `RAW_LOG_BACKED`.

## Raw-log discipline

### Canonical raw artifact

For each experiment, choose one canonical raw artifact.

Examples:

```text
CSV summary for deterministic sweeps
JSON for nested metrics
JSONL for event-level logs
NPZ plus JSON metadata for large arrays
```

The report must name the canonical raw artifact.

### Regeneration invariant

For deterministic experiments:

```text
fresh run from committed script == committed canonical raw log
```

This should be enforced by `scripts/check_raw_logs.py` and CI.

### Do not commit stale raw logs

If script behavior changes, regenerate raw logs and update reports in the same change set.

### Do not rely on Markdown as data

Markdown is for explanation. Data lives in `data/`.

## CI / pre-commit policy

At minimum, add GitHub Actions:

```yaml
name: Reproducibility
on:
  push:
  pull_request:
jobs:
  raw-log-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Install dependencies
        run: pip install -r requirements.txt
      - name: Check RAW_LOG_BACKED artifacts
        run: python scripts/check_raw_logs.py
```

Optional local pre-commit command:

```bash
python scripts/check_raw_logs.py
```

## File naming conventions

Use stable names.

```text
scripts/<layer>/<experiment_name>.py
data/<experiment_name>/<experiment_name>_seed<seed>.<json|csv|jsonl>
results/<experiment_name>_YYYY-MM-DD.md
experiments/<experiment_name>_protocol_YYYY-MM-DD.md
```

Prefer lowercase snake_case.

## Commit conventions

Use compact, factual commit messages.

Examples:

```text
Add converter core generator
Add converter raw CSV log
Add converter report
Register converter status
Refresh raw logs after generator change
Quarantine unverified phase transport claim
```

Avoid commit messages that claim success unless success is code-backed.

## Agent operating rules

### Before running experiments

Read:

```text
README.md
SKILLS.md
results/STATUS.md
relevant protocol in experiments/
existing generator script if extending one
```

### During experiments

Keep notes of:

```text
exact command
seed
config
runtime assumptions
unexpected failures
control results
```

### After experiments

Commit or update:

```text
generator script
raw log
report
STATUS.md
check_raw_logs.py if RAW_LOG_BACKED
```

### Never do this

```text
Do not paste GPT-made tables into results/ as if they are raw results.
Do not call a result quantum-specific without classical controls.
Do not leave stale logs after changing a script.
Do not mark a report RAW_LOG_BACKED without raw-log verification.
Do not hide raw json/csv in .gitignore.
Do not mix component-building claims and quantum-witness claims.
Do not use life-like labels such as metabolism/self-repair/self-regulation unless directly measured and defined.
```

## Component-development workflow

For classical-effective information-material parts, the goal is to build a reusable part table.

Each component should have:

```text
role
state variables
inputs
outputs
control parameters
material-property curves
failure modes
composition interfaces
```

Example roles:

```text
membrane:
  location/access: filters and separates

source-sink:
  quantity: injects and removes material

reservoir:
  timing: stores, delays, leaks, overflows

converter:
  identity/meaning: maps A to P/Q/Pprime with fidelity and drift

road:
  transport: biases future movement or arrival

buffer:
  stability: smooths flux or stress
```

Component reports should avoid quantum/audit vocabulary unless explicitly framed as future compatibility.

## Quantum/audit workflow

Use audit experiments only when making or testing quantum-specific claims.

Quantum/audit reports must include:

```text
quantum witness definition
classical probability control
classical complex-wave control if phase is involved
marginal/population matching audit where needed
dephase or measurement control where relevant
failure criteria
```

Accept negative results. A result showing that a behavior is classical-effective is valuable and should be recorded.

## Quarantine workflow

If a claim lacks code/log support or fails reproduction:

```text
1. move or mark it as QUARANTINED_CLAIM in results/STATUS.md
2. edit README so it is not listed as valid claim
3. keep the old report only as historical note if useful
4. create a protocol for how to retest it
```

Do not delete failed results unless they contain private data or serious mistakes that would mislead. Prefer labeling over erasing.

## Paper and Zenodo preparation

The final paper should cite the GitHub repository for code and reproducibility. Zenodo can archive a release snapshot.

Before Zenodo release:

```text
1. all main claims are CODE_BACKED or RAW_LOG_BACKED
2. results/STATUS.md is complete
3. scripts/check_raw_logs.py passes
4. README states valid claims and limitations
5. requirements.txt is complete
6. raw logs for main figures are committed or archived
7. figure-generation scripts exist
8. paper/ contains manuscript and figure/table mapping
9. create a GitHub release tag, e.g. v0.1-paper
10. archive that release on Zenodo and record DOI in README
```

Paper figure/table mapping should look like:

```text
paper/Figure_1:
  script: scripts/...
  raw: data/...
  report: results/...

paper/Table_1:
  script: scripts/...
  raw: data/...
  report: results/...
```

## Empty repository bootstrapping checklist

When starting from an empty repo, first create:

```text
README.md
SKILLS.md
requirements.txt
.gitignore
results/STATUS.md
data/README.md
experiments/README.md
scripts/check_raw_logs.py
.github/workflows/reproducibility.yml
```

Then create the first experiment in this order:

```text
1. experiments/<name>_protocol_YYYY-MM-DD.md
2. scripts/<name>.py
3. data/<name>/<name>_seed0.csv or json
4. results/<name>_YYYY-MM-DD.md
5. update results/STATUS.md
6. update scripts/check_raw_logs.py
7. run CI or local check
```

## Handoff prompt for another agent

Use this when asking another agent to continue work:

```text
You are working in this repository as a reproducibility-focused research agent.
Read README.md, SKILLS.md, and results/STATUS.md before editing.
Do not create result claims without committed scripts and raw logs.
Keep classical-effective component work separate from quantum/audit witness work.
If you create or update a RAW_LOG_BACKED result, update scripts/check_raw_logs.py and ensure it passes.
Do not paste unreproducible tables into results/.
Prefer clear negative results and quarantined claims over overclaiming.
```

## Final reminder

```text
A repository intended for Zenodo is not a notebook scratchpad.
It is a public chain of evidence.
```

Keep the chain intact:

```text
protocol -> script -> raw log -> report -> status -> CI -> paper -> Zenodo
```
